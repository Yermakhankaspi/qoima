# QOIMA — Страница «Показатели»

Готовые файлы для Next.js (App Router). Подключаются к Kaspi Merchant API
через переменную окружения `KASPI_API_TOKEN`.

## Структура файлов

```
app/
├── api/
│   └── pokazateli/
│       └── route.ts              ← API endpoint (GET /api/pokazateli)
└── pokazateli/
    ├── page.tsx                  ← Страница «Показатели»
    └── pokazateli.module.css     ← Стили (тёмная тема, оранжевый акцент)

lib/
└── kaspi.ts                      ← Утилиты для запросов к Kaspi API
```

## Установка

### 1. Скопировать файлы в проект

Положите файлы в указанные пути относительно корня вашего Next.js проекта.

### 2. Проверить алиас `@/lib`

В `tsconfig.json` должно быть:

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./*"]
    }
  }
}
```

Если у вас другая структура — замените в `app/api/pokazateli/route.ts`
строку `from '@/lib/kaspi'` на относительный путь, например `from '../../../lib/kaspi'`.

### 3. Переменная окружения

В Vercel → Settings → Environment Variables должна быть:

```
KASPI_API_TOKEN = <ваш токен из Kaspi Merchant Cabinet>
```

Токен берётся в кабинете Kaspi → Настройки → API.

### 4. Поменять навигацию

В вашем сайдбаре переименуйте пункт **«Дашборд»** на **«Показатели»**
и направьте ссылку на `/pokazateli`.

## Что считается

Endpoint `/api/pokazateli` забирает заказы за последние 60 дней и считает:

- **Сегодня / Неделя / Месяц** — количество выданных заказов, выручка, разница с предыдущим периодом
- **График** — выручка по дням текущей и прошлой недели (сравнение)
- **Просроченные** — заказы со статусом `ACCEPTED_BY_MERCHANT`, созданные больше 24 часов назад (Kaspi штрафует за просрочку отправки)
- **Возвраты** — заказы со статусом `RETURNED`

## Что важно знать про Kaspi API

- **Лимит запросов**: Kaspi ограничивает до ~100 запросов в минуту на токен.
  Сейчас на один запрос страницы тратится 1-3 запроса к Kaspi (пагинация).
  Если у вас > 100 заказов в день — добавьте кэш (Vercel KV / Upstash Redis).

- **Дата**: Kaspi отдаёт `creationDate` в миллисекундах (unix timestamp).

- **Статусы заказа** (`status`):
  - `APPROVED_BY_BANK` — оплачен, но ещё не принят продавцом
  - `ACCEPTED_BY_MERCHANT` — принят, на упаковке
  - `COMPLETED` — выдан покупателю (идёт в выручку)
  - `CANCELLED` — отменён
  - `RETURNED` — возврат

- **Состояния** (`state`): `NEW`, `SIGN_REQUIRED`, `PICKUP`, `DELIVERY`,
  `KASPI_DELIVERY`, `ARCHIVE`.

## Следующие шаги

1. **Кэш**: добавьте Vercel KV / Redis чтобы не дёргать Kaspi на каждый запрос.
   Один cron раз в 10 минут → пишет в KV → страница читает из KV.

2. **Себестоимость**: Kaspi не знает закупочную цену.
   Создайте таблицу `cost_prices` (SKU → закупочная цена) в Vercel Postgres,
   чтобы считать **чистую прибыль** = выручка − себестоимость − комиссия Kaspi (~12-15%).

3. **Расходы**: страница для ручного ввода расходов (реклама, аренда, зп) —
   тогда сможете показать полный P&L отчёт.

4. **Деньги / остаток**: эндпоинт `/payments/balance` есть не у всех тарифов
   Kaspi. Если нет — придётся синхронизировать вручную или через банк-API.
